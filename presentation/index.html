The Problem with Today's Setup

✅ OpenCode is great for…
  • Individual developers — fast, interactive, terminal-native
  • One-shot tasks: "write this function", "fix this bug"
  • Bash + file tools built-in, familiar workflow
  • OpenCode serve + TUI = powerful remote pair
⚠ But when tasks grow beyond the terminal…
  • Multi-step, cross-session tasks lose state on exit
  • No structured plan visible to reviewers
  • Files dumped into context → window overflows
  • No pause-and-wait for human approval mid-task
  • Cannot deploy to K8s / LangFlow / Foundry
Goal: keep your OpenCode skills for solo terminal work — add Deep Agents for orchestrated, stateful, team-reviewable, platform-deployable tasks.

Configuring the LLM Gateway

One place to set. Everything — orchestrator, all skills, all sub-agents — reads from here.

# agents/config.py  ← the single source of truth

from langchain_openai import ChatOpenAI   # OpenAI-compatible API

# ── LLM Gateway endpoint ────────────────────────────────────────
LLM_GATEWAY_URL = "https://llm-gateway.your-org.internal/v1"
LLM_GATEWAY_KEY = os.environ["LLM_GATEWAY_API_KEY"]   # set in env / K8s secret

# ── Model aliases — change these to whatever the gateway exposes ─
MODEL_GENERAL = "general"    # general-purpose reasoning
MODEL_CODE    = "coder"      # code-optimized model
MODEL_FAST    = "fast"       # lightweight, low-latency

def make_llm(model: str = MODEL_GENERAL, temperature: float = 0.1):
    """Return a LangChain ChatOpenAI pointed at the internal gateway."""
    return ChatOpenAI(
        model=model,
        temperature=temperature,
        base_url=LLM_GATEWAY_URL,
        api_key=LLM_GATEWAY_KEY,
    )
Any OpenAI-compatible gateway works: LiteLLM proxy, Azure OpenAI, vLLM, TGI. If the gateway uses a different SDK, swap ChatOpenAI for the matching LangChain integration — make_llm() is the only place to change.

Without it — repeated boilerplate everywhere

        temperature=0.1,
    )  # ← repeated in every skill
    ...

@tool
def write_code(task: str) -> str:
    llm = ChatOpenAI(
        model="coder",
        base_url="https://llm-gateway.../v1",  # ← again
        api_key=os.environ["LLM_GATEWAY_API_KEY"],
        temperature=0.1,
    )
Change the gateway URL → edit every skill file.
With make_llm() — one place to change 
# skills.py — each skill is now one line
@tool
def analyze_data(data: str) -> str:
    llm = make_llm(MODEL_GENERAL)   # ← that's it
    ...

@tool
def write_code(task: str) -> str:
    llm = make_llm(MODEL_CODE)      # ← different model,
    ...                             #   same gateway config

Why skills each call make_llm(): every skill may need a different model — analysis vs. coding vs. fast lookup. The orchestrator chooses which skill to call; the skill chooses which model to use. Separation of concerns.

How Skills Call make_llm() — The Flow

Each skill call is isolated — its own model, its own context window, returns a plain string

Orchestrator LLM (MODEL_GENERAL) │ "I need to analyze data AND write code" │ ├─► analyze_data("sales=120, users=80") │ └─ make_llm(MODEL_GENERAL).invoke(...) ← separate context window │ returns: "Revenue up 12%, risk in segment B" │ ├─► write_code("bar chart for sales trend") │ └─ make_llm(MODEL_CODE).invoke(...) ← separate context window │ returns: "import matplotlib..." │ └─ "Here is the analysis and the code: ..." ← orchestrator composes final answer
Context isolation
Each skill call has its own clean context — no prior conversation leaks in.
Model specialization
Right model per job. Coder model never sees analysis prompts.
make_llm() is not LangChain API
It's our custom factory wrapping ChatOpenAI. Official docs: langchain_openai.ChatOpenAI

Advanced: Recursive LLM Calls Inside a Skill

When input is too large for one call — split, process each chunk, synthesize

analyze_large_dataset(huge_data) │ ├─ chunk 1 → make_llm().invoke() ← small context │ returns: "segment 1 summary" ├─ chunk 2 → make_llm().invoke() ← small context │ returns: "segment 2 summary" ├─ chunk 3 → make_llm().invoke() ← small context │ returns: "segment 3 summary" │ └─ make_llm().invoke(summaries) ← synthesize returns: full report string
Orchestrator sees one tool call and one result string. It never knows the skill made multiple LLM calls internally. 
@tool
def analyze_large_dataset(data: str) -> str:
    """Handles datasets too large for one context."""
    chunks = _split_by_tokens(data, max_tokens=3000)

    # Each chunk → its own small LLM call
    partials = [
        make_llm(MODEL_GENERAL).invoke([
            SystemMessage("Analyze this segment briefly."),
            HumanMessage(chunk),
        ]).content
        for chunk in chunks
    ]

    # Final synthesis — only summaries, not raw data
    return make_llm(MODEL_GENERAL).invoke([
        SystemMessage(
            "Synthesize these analysis segments "
            "into one concise report."
        ),
        HumanMessage("\n---\n".join(partials)),
    ]).content
This is not recursive in the strict CS sense — no function calls itself. It is chained parallel LLM calls (map) followed by one synthesis call (reduce). Each call uses only a fraction of the context window. The parent orchestrator's context is completely unaffected.

Virtual File System (VFS) — The Core Idea

Without VFS — the context problem
Context window (128k tokens) ┌────────────────────────────────────┐ │ System prompt ~500 tok │ │ User: "write a report" ~30 tok │ │ Tool result: analysis ~800 tok │ │ FILE: data.csv ~2000 tok │ ← !! │ FILE: report.txt ~3000 tok │ ← !! │ FILE: chart.py ~1500 tok │ ← !! │ Agent reasoning ~400 tok │ │ ... more tool calls ... │ │ ↑ CONTEXT FULL ↑ │ └────────────────────────────────────┘ Agent can no longer reason. Task fails or hallucinates.
With VFS — files never enter context
Context window (128k tokens) ┌────────────────────────────────────┐ │ System prompt ~500 tok │ │ User: "write a report" ~30 tok │ │ Tool result: analysis ~800 tok │ │ write_file("data.csv") ~10 tok │ ← ref only │ write_file("report.txt") ~10 tok │ ← ref only │ write_file("chart.py") ~10 tok │ ← ref only │ Agent reasoning ~400 tok │ │ │ │ ← PLENTY OF SPACE LEFT → │ └────────────────────────────────────┘ Agent stays sharp across many tasks.

VFS — How It Works

# Per-thread, in-memory dict (swap for Redis / S3)
_vfs: dict[str, dict[str, str]] = defaultdict(dict)
# key = thread_id → { path → content }

@tool
def write_file(path: str, content: str,
               config: RunnableConfig) -> str:
    """Write content to a virtual file."""
    tid = config["configurable"]["thread_id"]
    _vfs[tid][path] = content
    return f"Written {len(content)} bytes to {path!r}"

@tool
def read_file(path: str,
              config: RunnableConfig) -> str:
    """Read a virtual file."""
    tid = config["configurable"]["thread_id"]
    return _vfs[tid].get(path, f"File not found: {path!r}")

@tool
def list_files(config: RunnableConfig) -> str:
    """List all files in the virtual file system."""
    tid = config["configurable"]["thread_id"]
    files = list(_vfs[tid].keys())
    return "\n".join(files) if files else "No files yet."
Thread isolation

Every conversation (thread) has its own namespace in _vfs. Concurrent users never see each other's files. The key is injected via RunnableConfig — LangGraph handles this automatically when you pass thread_id in the config.
Production swap

The in-memory dict is a demo store. Replace the two lines that read/write _vfs[tid] with any key/value backend: Redis, S3, Postgres jsonb, or a proper blob store. No other code changes needed.

VFS — Real Workflow

User: "Analyze sales data, write a Python visualisation, explain results" Step 1 → write_todos(["analyze data","write code","write summary"]) todos visible to reviewers immediately Step 2 → analyze_data("sales=120, users=80, churn=5%") returns: "Revenue up 12%, churn risk in segment B..." write_file("analysis.txt", <analysis result>) ← NOT in context Step 3 → write_code("visualise sales trend as bar chart") returns: <300 lines of Python> write_file("chart.py", <code>) ← NOT in context Step 4 → request_options("Run chart.py or refine first?", ["Run it now", "Refine the analysis"]) PAUSED — waits for human Step 5 → Human picks "Run it now" → Command(resume="Run it now") Agent reads chart.py path from state, calls run tool write_file("chart.png", <binary>) Context window throughout: ~2,000 tokens total. Never overflows.
The agent never holds file content in messages — only paths. Full content is in VFS, retrieved on demand by tools or the consumer.

Deep Agent — State & Graph

State TypedDict 
class DeepAgentState(TypedDict):
    # Append-only message list
    # LangGraph manages reduction
    messages: Annotated[list, add_messages]

    # Published by write_todos tool
    # Visible to clients in real time
    todos: list[str]

    # Index of paths in VFS
    # Content stored off-graph
    vfs_keys: list[str]
Why custom StateGraph?

create_react_agent gives you messages only. Custom StateGraph adds todos and vfs_keys as first-class state — they persist across interrupts, survive time-travel replay, and are queryable from GET /threads/{id}/state.
Graph structure: ┌────────────────────┐ │ agent_node │◄─────────────────┐ │ │ │ │ LLM with all │ │ │ tools bound │──tool calls──► │ │ │ │ │ No tools called? │──► END │ └────────────────────┘ │ │ │ ┌─────────▼──────────┐ │ │ tools_node │──────────────────┘ │ │ │ ToolNode executes │ updated state │ all pending calls │ (messages + todos │ │ + vfs_keys) └────────────────────┘
The loop runs until the LLM produces a response with no tool calls. HITL interrupt breaks out of the loop mid-execution.

Each skill calls make_llm() with the appropriate model alias.

@tool

def search_knowledge(query: str) -> str:

Orchestrator (general LLM) picks which skill to call. Skill runs its own LLM call in isolation. Results flow back as tool output — never as file content in context.

Human-in-the-Loop

from langgraph.types import interrupt, Command

@tool
def request_options(question: str,
                    options: list[str],
                    config: RunnableConfig) -> str:
    # Execution pauses here.
    # Full graph state saved to checkpointer.
    # Thread returns "interrupted" status.
    choice = interrupt({
        "question": question,
        "options":  options,
    })
    return f"Human chose: {choice!r}"

# Resume — from any process, any time:
agent.invoke(
    Command(resume="Full report"),
    config={"configurable": {
        "thread_id": "proj-42"
    }}
)
Agent running... │ ┌───────▼────────────┐ │ request_options │ │ "Choose depth" │ │ [A] Summary │ │ [B] Full report │ └───────┬────────────┘ │ interrupt() │ ← state frozen in checkpointer │ ← HTTP response: "interrupted" │ human reviews plan, picks option B │ │ Command(resume="Full report") ┌───────▼────────────┐ │ Agent resumes │ │ exact state │ │ choice injected │ └────────────────────┘
Resume can come from a different process, hours later. All VFS files, todos, and message history are intact.

Hosting LangGraph Agents Inside Deep Agents

Any compiled LangGraph graph can be wrapped as a tool and given to a parent agent. The parent treats it like any other tool — it receives a task string, runs the sub-graph fully (including its own tool loops), and returns a result string back to the parent. 
from langgraph.prebuilt import create_react_agent

# Sub-agent with its own specialised tools
CODE_AGENT = create_react_agent(
    make_llm(MODEL_CODE),
    tools=[write_file, read_file, list_files],
    checkpointer=MemorySaver(),
)

@tool
def delegate_coding(task: str, config: RunnableConfig) -> str:
    """Delegate a full coding task to the specialist code agent."""
    tid = config["configurable"]["thread_id"]
    result = CODE_AGENT.invoke(
        {"messages": [{"role": "user", "content": task}]},
        # sub-agent gets its own thread namespace
        config={"configurable": {"thread_id": f"{tid}:code"}},
    )
    return result["messages"][-1].content

# Give this tool to the Deep Agent orchestrator
DEEP_AGENT = build_deep_agent(extra_tools=[delegate_coding])

Sub-Agent Patterns

Specialist delegation

Orchestrator handles planning + HITL. Delegates coding, analysis, or search to dedicated sub-graphs configured with the right model and tools.
Parallel fan-out

LangGraph Send() runs multiple sub-agents in parallel. Each writes results to VFS. Orchestrator synthesises when all finish.
Recursive depth

Sub-agent can itself have sub-agents. Each level gets a unique thread-id suffix — checkpointers don't collide. Depth limited by latency budget.
Deep Agent (orchestrator · general LLM · all tools) ├── write_todos / write_file / request_options ← native tools ├── analyze_data → inline skill (same process, general LLM) ├── delegate_coding → CODE_AGENT sub-graph (code LLM) └── delegate_search → SEARCH_AGENT sub-graph (general LLM + Glean)

Deep Agent in a Notebook or CI Pipeline

# Works in Jupyter, VS Code notebook, or a CI script
import asyncio
from agents.deep_agent import DEEP_AGENT
from agents.config import make_llm  # uses LLM gateway automatically

thread = "notebook-demo-01"
config = {"configurable": {"thread_id": thread}}

# ── Run and collect ─────────────────────────────────────────────
tokens = []
tool_calls = []

async def run():
    async for event in DEEP_AGENT.astream_events(
        {"messages": [{"role": "user",
            "content": "Write a Python function to parse a CSV and compute averages"}]},
        config=config,
        version="v2",
    ):
        if event["event"] == "on_chat_model_stream":
            tokens.append(event["data"]["chunk"].content)
        elif event["event"] == "on_tool_end":
            tool_calls.append(event["name"])

asyncio.run(run())

print("".join(tokens))          # final answer
print("Tools used:", tool_calls)

# ── Read VFS output ─────────────────────────────────────────────

print(get_vfs(thread))          # {"solution.py": "def parse_csv(..."})

🌊 LangFlow — Custom Component Wrapper

How it works

LangFlow has no built-in "import LangGraph graph" node. You write a Component subclass that wraps your compiled graph. It appears in LangFlow's canvas as a draggable block with typed inputs/outputs. No-code users wire it visually; developers control the graph internals.
LLM gateway in LangFlow

The component calls make_llm() internally — reads LLM_GATEWAY_API_KEY from environment. Set once in LangFlow's secrets store; all components inherit it. 
from langflow.custom import Component
from langflow.io import MessageTextInput, Output

class DeepAgentComponent(Component):
    display_name = "Deep Agent"
    description  = "LangGraph Deep Agent — VFS + Skills + HITL"

    inputs = [
        MessageTextInput(
            name="message",
            display_name="User Message"
        ),
    ]
    outputs = [
        Output(display_name="Response",
               name="response",
               method="run_agent"),
    ]

    def run_agent(self) -> Message:
        result = DEEP_AGENT.invoke(
            {"messages": [{"role": "user",
                           "content": self.message}]},
            config={"configurable": {
                "thread_id": self.session_id
            }},
        )
        return Message(text=result["messages"][-1].content)

🔍 Glean — Search Tool Inside Deep Agent

Glean cannot host the Deep Agent
No stateful graph execution, no VFS, no HITL. Glean Agents API manages only Glean-internal agents — not external LangGraph graphs.
What Glean provides (via glean-agent-toolkit)

glean_search.as_langchain_tool() — official adapter converts Glean search into a LangChain-compatible tool. Drop it directly into the Deep Agent's tool list. 
from glean.agent_toolkit.tools import glean_search

langchain_tool = glean_search.as_langchain_tool()
DEEP_AGENT = build_deep_agent(
    extra_tools=[langchain_tool]
)
Or call Glean API directly as a skill 
@tool
def search_knowledge(query: str) -> str:
    """Search enterprise knowledge via Glean."""
    resp = requests.post(
        "https://your-org.glean.com/api/rest/search",
        headers={"Authorization": f"Bearer {GLEAN_TOKEN}"},
        json={"query": query, "pageSize": 5},
    )
    hits = resp.json().get("results", [])
    docs = "\n\n".join(
        h["document"]["title"] + "\n" +
        h["snippets"][0]["text"] for h in hits
    )
    return make_llm(MODEL_GENERAL).invoke(
        f"Answer based on:\n{docs}\n\nQ: {query}"
    ).content
Glean = enterprise search index. Deep Agent = reasoning and orchestration on top. Results written to VFS — never to context window.

Mode A — FastAPI POST /chat/stream

Server side — main.py 
@app.post("/chat/stream")
def chat_stream(request: Request, body: ChatStreamRequest):
    # Creates/reuses thread internally
    tid = body.thread_id or str(uuid.uuid4())
    if not thread_get(tid):
        thread_create(tid, {})
    # Translates Aegra named events → flat type schema
    async def generator():
        async for chunk in _stream_run(agent, tid, ...):
            ev = chunk["event"]
            if ev == "messages/partial":
                yield {"data": json.dumps({"type":"token","content":...})}
            elif ev == "values":
                yield {"data": json.dumps({"type":"vfs_update","vfs":...})}
            elif ev == "end":
                yield {"data": json.dumps({"type":"done"})}
    return EventSourceResponse(generator())
React GUI — runFastAPI() 
function runFastAPI(message, h) {
  // Uses mode's stable sessionId as thread_id
  // → same thread reused across messages
  streamFetch("/api/chat/stream", "POST",
    JSON.stringify({ message, thread_id: bucket.serverThreadId }),
    function(evName, data) {
      if (data.type === "token")
        h.appendToken(data.content);
      else if (data.type === "tool_start")
        h.pushMsg({ role:"tool", ... });
      else if (data.type === "vfs_update")
        h.setVfs(v => Object.assign({}, v, data.vfs));
      else if (data.type === "done")

    }

}
Flat schema — simplest to parse. Server owns the thread lifecycle.

Mode B — Aegra LangGraph Platform Spec

SSE event stream from /threads/{id}/runs/stream: event: metadata data: {"run_id":"…","assistant_id":"deep"} event: updates data: {"node":"agent","run_id":"…"} event: messages/partial data: {"content":"Hello","type":"AIMessageChunk"} event: tool_start data: {"tool":"write_file","input":{…}} event: tool_end data: {"tool":"write_file","output":"Written…"} event: values data: {"messages":[…],"vfs":{…},"todos":[…]} event: end data: {}
React GUI — runAegra() 
function runAegra(message, h) {
  // First message → create server thread, cache it
  // Subsequent messages → reuse same thread
  // (conversation + VFS persist on server)
  var existing = bucket.serverThreadId;
  function doRun(tid) {
    streamFetch("/api/threads/"+tid+"/runs/stream",
      "POST", JSON.stringify({assistant_id:"deep",
        input:{messages:[{role:"user",content:message}]}}),
      function(evName, data) {
        if (evName==="messages/partial") h.appendToken(data.content);
        if (evName==="values") {
          h.setVfs(data.vfs);
          h.setTodos(data.todos);
        }
        if (evName==="interrupt") setHitl({...data, mode:"aegra"});
        if (evName==="end") { h.finalizeAI(); setBusy(false); }
      }
    );
  }
  existing ? doRun(existing) : createThread().then(doRun);
}
Named events — caller owns thread lifecycle, inspects state via GET /threads/{id}/state.

Mode C — Langflow Flow Orchestration

GUI → POST /langflow/api/v1/run/{flow_id}?stream=true │ Langflow flow: ChatInput → AegraDeepAgent → ChatOutput │ Component.run_agent() │ POST /threads/{id}/runs/stream (Aegra SSE) │ iter SSE → on "messages/partial" chunk arrives │ self._event_manager.on_token(data={"chunk": chunk}) │ Langflow EventManager puts {"event":"token",...} into the async SSE queue │ GUI receives: event: token │ data: {"chunk":"Hello"} ◄┘ ← live, token by token
Thread isolation
Each Langflow session creates an Aegra thread tagged {"lf_session": session_id}. After run, GUI calls GET /sessions/{id}/state → VFS + todos.
Same Deep Agent
Langflow is just another caller of the Aegra API. The agent, VFS, tools, and PostgreSQL backend are identical to FastAPI and Aegra modes.
No-code extensible
The flow lives in Langflow's canvas. Add preprocessing, routing, or parallel calls visually — without changing the Deep Agent code.

Live Streaming from a Langflow Custom Component

Common misconception: "Custom Langflow components can't stream tokens — only LangChain LLM nodes can."
Reality: Langflow injects self._event_manager on every component before execution. Calling on_token() on it is the correct internal API — the same one Langflow's own LLM nodes use.
Before — sync component, no streaming 
def run_agent(self) -> Message:
    tokens = []
    for token in self._stream_tokens(...):
        tokens.append(token)   # collect silently
    return Message(text="".join(tokens))
    # ↑ Langflow emits add_message AFTER this returns
    # GUI sees: long silence → full response at once
After — forward each token into Langflow's SSE queue 
def run_agent(self) -> Message:
    tokens = []
    for token in self._stream_tokens(...):
        tokens.append(token)
        # Langflow injected this before calling run_agent()
        if self._event_manager:
            self._event_manager.on_token(
                data={"chunk": token, "id": message_id}
            )
            # ↑ puts {"event":"token","data":{…}} into
            #   the async SSE queue immediately
    return Message(text="".join(tokens))
    # GUI sees: live tokens as they arrive ✓
EventManager.on_token() calls self._loop.call_soon_threadsafe(queue.put_nowait, item) — correctly crossing from the sync component thread into the async event loop. Not a hack. This is Langflow's own internal event pipeline.

React GUI — Per-Mode State Isolation

Problem — shared state pollutes modes 
// Switching FastAPI → Aegra showed FastAPI's messages
// Tool results from one mode appeared in another
const [todos, setTodos] = useState([]);
const [vfs,   setVfs]   = useState({});
const [threadId] = useState(() => genId()); // shared!
Solution — per-mode data buckets in a ref 
const modeDataRef = useRef({
  fastapi:  { sessionId, msgs:[], raw:[], todos:[], vfs:{}, serverThreadId:null },
  aegra:    { sessionId, msgs:[], raw:[], todos:[], vfs:{}, serverThreadId:null },
  langflow: { sessionId, msgs:[], raw:[], todos:[], vfs:{}, serverThreadId:null },
});
// Render reads from modeDataRef.current[serverMode]
// Switching modes instantly shows that mode's history
makeHelpers(mode) — captured-mode closures 
function makeHelpers(mode) {
  function s() { return modeDataRef.current[mode]; }
  // All writes go to the captured mode's bucket
  // even if user switches mode mid-run
  function pushMsg(m)  { s().msgs = [...s().msgs, m]; forceRender(); }
  function appendToken(c) { /* update last msg in s() */ }
  function setVfs(u) { s().vfs = typeof u==="function" ? u(s().vfs) : u; forceRender(); }
  function setTodos(t) { s().todos = t; forceRender(); }
  return { pushMsg, appendToken, setVfs, setTodos, finalizeAI, pushRaw };
}
// Used as:
var h = makeHelpers(serverMode);  // capture before async starts
h.pushMsg({ role:"user", content: msg });
runFastAPI(msg, h);  // h is closed over correct mode
React 18 automatic batching (concurrent mode) suppressed intermediate renders inside Promise callbacks. Fixed with ReactDOM.flushSync() per SSE chunk.

VFS Integration Across All Three Modes

PostgreSQL (agentdb) vfs_files table: thread_id | path | content | updated_at threads table: thread_id | metadata (JSONB) | status agent_todos: thread_id | todos (JSONB) FastAPI mode: thread_id = modeData.fastapi.sessionId → values event carries full vfs dict at run end → GUI updates Files panel directly from SSE stream Aegra mode: thread_id = modeData.aegra.serverThreadId (created on first message) → values event carries full vfs dict + todos at run end → GUI updates Files + Plan panels directly from SSE stream Langflow mode: thread_id = Aegra thread tagged {"lf_session": "lf-<sessionId>"} → after event: end → GUI calls GET /api/sessions/{sessionId}/state → server: SELECT thread WHERE metadata->>'lf_session' = $1 → returns vfs_get_all(thread_id) + todos_get(thread_id)
All three modes write to the same PostgreSQL VFS table. Switching from Langflow to Aegra with the same session doesn't share VFS — each mode has its own isolated thread and its own Files panel.
