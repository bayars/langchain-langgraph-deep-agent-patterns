The Problem with Today's Setup

⚠ Fragmented Toolchain
  • Team split: LangGraph for loops vs OpenCode for terminal skills
  • Raw FastAPI glue between them
  • Separate memory, context, threading per tool
  • No shared state between sessions
⚠ OpenCode Limits
  • Great at a developer terminal — not production-ready
  • No persistent thread state across tasks
  • No structured planning — humans verify nothing
  • Context window fills with file content
  • Can't pause mid-task for human input
Goal: one harness that replaces OpenCode for shared tasks, adds state + planning + specialised LLMs + human oversight, and deploys to any platform.

Why Deep Agents Beat OpenCode

Capability OpenCode / raw scripting Deep Agent (LangGraph)
Persistent thread state❌ lost on process exit✅ saved to checkpointer
Structured planning❌ implicit in prompt only✅ write_todos → published plan
File management❌ dumps files into context✅ VFS — files never touch context window
Multi-model routing❌ one model for everything✅ right model per task (skills bus)
Human-in-the-loop❌ ctrl-C and re-run✅ interrupt() → choice → resume
Audit / replay❌ log file only✅ full state history in checkpointer
Platform deployment❌ developer laptop only✅ K8s + Aegra, LangFlow, Foundry
Nested sub-agents❌ manual piping✅ any graph is a tool

Configuring the LLM Gateway

One place to set. Everything — orchestrator, all skills, all sub-agents — reads from here.

# agents/config.py  ← the single source of truth

from langchain_openai import ChatOpenAI   # OpenAI-compatible API

# ── LLM Gateway endpoint ────────────────────────────────────────
LLM_GATEWAY_URL = "https://llm-gateway.your-org.internal/v1"
LLM_GATEWAY_KEY = os.environ["LLM_GATEWAY_API_KEY"]   # set in env / K8s secret

# ── Model aliases — change these to whatever the gateway exposes ─
MODEL_GENERAL = "general"    # general-purpose reasoning
MODEL_CODE    = "coder"      # code-optimized model
MODEL_FAST    = "fast"       # lightweight, low-latency

def make_llm(model: str = MODEL_GENERAL, temperature: float = 0.1):
    """Return a LangChain ChatOpenAI pointed at the internal gateway."""
    return ChatOpenAI(
        model=model,
        temperature=temperature,
        base_url=LLM_GATEWAY_URL,
        api_key=LLM_GATEWAY_KEY,
    )
Any OpenAI-compatible gateway works: LiteLLM proxy, Azure OpenAI, vLLM, TGI. If the gateway uses a different SDK, swap ChatOpenAI for the matching LangChain integration — make_llm() is the only place to change.

Every Skill Uses make_llm()

Skills are just tools that call make_llm() — each gets the right model for its job.

# agents/skills.py

    """Analyze data and return structured insights."""

Virtual File System (VFS) — The Core Idea

Without VFS — the context problem
Context window (128k tokens) ┌────────────────────────────────────┐ │ System prompt ~500 tok │ │ User: "write a report" ~30 tok │ │ Tool result: analysis ~800 tok │ │ FILE: data.csv ~2000 tok │ ← !! │ FILE: report.txt ~3000 tok │ ← !! │ FILE: chart.py ~1500 tok │ ← !! │ Agent reasoning ~400 tok │ │ ... more tool calls ... │ │ ↑ CONTEXT FULL ↑ │ └────────────────────────────────────┘ Agent can no longer reason. Task fails or hallucinates.
With VFS — files never enter context
Context window (128k tokens) ┌────────────────────────────────────┐ │ System prompt ~500 tok │ │ User: "write a report" ~30 tok │ │ Tool result: analysis ~800 tok │ │ write_file("data.csv") ~10 tok │ ← ref only │ write_file("report.txt") ~10 tok │ ← ref only │ write_file("chart.py") ~10 tok │ ← ref only │ Agent reasoning ~400 tok │ │ │ │ ← PLENTY OF SPACE LEFT → │ └────────────────────────────────────┘ Agent stays sharp across many tasks.

VFS — How It Works

# Per-thread, in-memory dict (swap for Redis / S3)
_vfs: dict[str, dict[str, str]] = defaultdict(dict)
# key = thread_id → { path → content }

@tool
def write_file(path: str, content: str,
               config: RunnableConfig) -> str:
    """Write content to a virtual file."""
    tid = config["configurable"]["thread_id"]
    _vfs[tid][path] = content
    return f"Written {len(content)} bytes to {path!r}"

@tool
def read_file(path: str,
              config: RunnableConfig) -> str:
    """Read a virtual file."""
    tid = config["configurable"]["thread_id"]
    return _vfs[tid].get(path, f"File not found: {path!r}")

@tool
def list_files(config: RunnableConfig) -> str:
    """List all files in the virtual file system."""
    tid = config["configurable"]["thread_id"]
    files = list(_vfs[tid].keys())
    return "\n".join(files) if files else "No files yet."
Thread isolation

Every conversation (thread) has its own namespace in _vfs. Concurrent users never see each other's files. The key is injected via RunnableConfig — LangGraph handles this automatically when you pass thread_id in the config.
Production swap

The in-memory dict is a demo store. Replace the two lines that read/write _vfs[tid] with any key/value backend: Redis, S3, Postgres jsonb, or a proper blob store. No other code changes needed.

VFS — Real Workflow

User: "Analyze sales data, write a Python visualisation, explain results" Step 1 → write_todos(["analyze data","write code","write summary"]) todos visible to reviewers immediately Step 2 → analyze_data("sales=120, users=80, churn=5%") returns: "Revenue up 12%, churn risk in segment B..." write_file("analysis.txt", <analysis result>) ← NOT in context Step 3 → write_code("visualise sales trend as bar chart") returns: <300 lines of Python> write_file("chart.py", <code>) ← NOT in context Step 4 → request_options("Run chart.py or refine first?", ["Run it now", "Refine the analysis"]) PAUSED — waits for human Step 5 → Human picks "Run it now" → Command(resume="Run it now") Agent reads chart.py path from state, calls run tool write_file("chart.png", <binary>) Context window throughout: ~2,000 tokens total. Never overflows.
The agent never holds file content in messages — only paths. Full content is in VFS, retrieved on demand by tools or the consumer.

Deep Agent — State & Graph

State TypedDict 
class DeepAgentState(TypedDict):
    # Append-only message list
    # LangGraph manages reduction
    messages: Annotated[list, add_messages]

    # Published by write_todos tool
    # Visible to clients in real time
    todos: list[str]

    # Index of paths in VFS
    # Content stored off-graph
    vfs_keys: list[str]
Why custom StateGraph?

create_react_agent gives you messages only. Custom StateGraph adds todos and vfs_keys as first-class state — they persist across interrupts, survive time-travel replay, and are queryable from GET /threads/{id}/state.
Graph structure: ┌────────────────────┐ │ agent_node │◄─────────────────┐ │ │ │ │ LLM with all │ │ │ tools bound │──tool calls──► │ │ │ │ │ No tools called? │──► END │ └────────────────────┘ │ │ │ ┌─────────▼──────────┐ │ │ tools_node │──────────────────┘ │ │ │ ToolNode executes │ updated state │ all pending calls │ (messages + todos │ │ + vfs_keys) └────────────────────┘
The loop runs until the LLM produces a response with no tool calls. HITL interrupt breaks out of the loop mid-execution.

Each skill calls make_llm() with the appropriate model alias.

@tool
def search_knowledge(query: str) -> str:
    """Search internal KB (Glean) and synthesise an answer."""
    hits = _call_glean_api(query)          # raw search results
    return make_llm(MODEL_GENERAL).invoke([

Orchestrator (general LLM) picks which skill to call. Skill runs its own LLM call in isolation. Results flow back as tool output — never as file content in context.

Human-in-the-Loop

from langgraph.types import interrupt, Command

@tool
def request_options(question: str,
                    options: list[str],
                    config: RunnableConfig) -> str:
    # Execution pauses here.
    # Full graph state saved to checkpointer.
    # Thread returns "interrupted" status.
    choice = interrupt({
        "question": question,
        "options":  options,
    })
    return f"Human chose: {choice!r}"

# Resume — from any process, any time:
agent.invoke(
    Command(resume="Full report"),
    config={"configurable": {
        "thread_id": "proj-42"
    }}
)
Agent running... │ ┌───────▼────────────┐ │ request_options │ │ "Choose depth" │ │ [A] Summary │ │ [B] Full report │ └───────┬────────────┘ │ interrupt() │ ← state frozen in checkpointer │ ← HTTP response: "interrupted" │ human reviews plan, picks option B │ │ Command(resume="Full report") ┌───────▼────────────┐ │ Agent resumes │ │ exact state │ │ choice injected │ └────────────────────┘
Resume can come from a different process, hours later. All VFS files, todos, and message history are intact.

Hosting LangGraph Agents Inside Deep Agents

Any compiled LangGraph graph can be wrapped as a tool and given to a parent agent. The parent treats it like any other tool — it receives a task string, runs the sub-graph fully (including its own tool loops), and returns a result string back to the parent. 
from langgraph.prebuilt import create_react_agent

# Sub-agent with its own specialised tools
CODE_AGENT = create_react_agent(
    make_llm(MODEL_CODE),
    tools=[write_file, read_file, list_files],
    checkpointer=MemorySaver(),
)

@tool
def delegate_coding(task: str, config: RunnableConfig) -> str:
    """Delegate a full coding task to the specialist code agent."""
    tid = config["configurable"]["thread_id"]
    result = CODE_AGENT.invoke(
        {"messages": [{"role": "user", "content": task}]},
        # sub-agent gets its own thread namespace
        config={"configurable": {"thread_id": f"{tid}:code"}},
    )
    return result["messages"][-1].content

# Give this tool to the Deep Agent orchestrator
DEEP_AGENT = build_deep_agent(extra_tools=[delegate_coding])

Sub-Agent Patterns

Specialist delegation

Orchestrator handles planning + HITL. Delegates coding, analysis, or search to dedicated sub-graphs configured with the right model and tools.
Parallel fan-out

LangGraph Send() runs multiple sub-agents in parallel. Each writes results to VFS. Orchestrator synthesises when all finish.
Recursive depth

Sub-agent can itself have sub-agents. Each level gets a unique thread-id suffix — checkpointers don't collide. Depth limited by latency budget.
Deep Agent (orchestrator · general LLM · all tools) ├── write_todos / write_file / request_options ← native tools ├── analyze_data → inline skill (same process, general LLM) ├── delegate_coding → CODE_AGENT sub-graph (code LLM) └── delegate_search → SEARCH_AGENT sub-graph (general LLM + Glean)

Deep Agent in a Notebook or CI Pipeline

# Works in Jupyter, VS Code notebook, or a CI script
import asyncio
from agents.deep_agent import DEEP_AGENT
from agents.config import make_llm  # uses LLM gateway automatically

thread = "notebook-demo-01"
config = {"configurable": {"thread_id": thread}}

# ── Run and collect ─────────────────────────────────────────────
tokens = []
tool_calls = []

async def run():
    async for event in DEEP_AGENT.astream_events(
        {"messages": [{"role": "user",
            "content": "Write a Python function to parse a CSV and compute averages"}]},
        config=config,
        version="v2",
    ):
        if event["event"] == "on_chat_model_stream":
            tokens.append(event["data"]["chunk"].content)
        elif event["event"] == "on_tool_end":
            tool_calls.append(event["name"])

asyncio.run(run())

print("".join(tokens))          # final answer
print("Tools used:", tool_calls)

# ── Read VFS output ─────────────────────────────────────────────
from agents.deep_agent import get_vfs
print(get_vfs(thread))          # {"solution.py": "def parse_csv(..."})

🌊 LangFlow — Native Agent Node

Deep Agent as a LangFlow component

Wrap the compiled graph in a Component class. It appears in LangFlow's canvas as a first-class draggable block. No-code users wire inputs and outputs visually. Developers edit the graph internals when needed.
LLM gateway in LangFlow

The component calls make_llm() internally — it reads LLM_GATEWAY_API_KEY from the environment. Set it once in LangFlow's secrets store; all agent components inherit it. 
from langflow.custom import Component
from langflow.io import MessageTextInput, Output

class DeepAgentComponent(Component):
    display_name = "Deep Agent"
    description  = "LangGraph Deep Agent — VFS + Skills + HITL"

    inputs = [
        MessageTextInput(
            name="message",
            display_name="User Message"
        ),
    ]
    outputs = [
        Output(display_name="Response",
               name="response",
               method="run_agent"),
    ]

    def run_agent(self) -> Message:
        result = DEEP_AGENT.invoke(
            {"messages": [{"role": "user",
                           "content": self.message}]},
            config={"configurable": {
                "thread_id": self.session_id
            }},
        )
        return Message(text=result["messages"][-1].content)

🔍 Glean — Search Tool, Not Agent Host

Why Glean can't host the Deep Agent

Glean is a search platform, not an agent runtime. It has no concept of stateful graph execution, VFS, tool loops, or HITL interrupts. There is no Glean API for running custom LangGraph graphs.
What Glean provides

Glean exposes a REST search API that returns ranked document snippets from your internal knowledge base. The Deep Agent calls it as one tool among many — results are synthesised by the LLM and written to VFS, not dumped into context. 
@tool
def search_knowledge(query: str) -> str:
    """Search enterprise knowledge via Glean."""
    resp = requests.post(
        "https://your-org.glean.com/api/rest/search",
        headers={"Authorization": f"Bearer {GLEAN_TOKEN}"},
        json={"query": query, "pageSize": 5},
    )
    hits  = resp.json().get("results", [])
    docs  = "\n\n".join(
        h["document"]["title"] + "\n" + h["snippets"][0]["text"]
        for h in hits
    )
    # Synthesise with general LLM from gateway
    return make_llm(MODEL_GENERAL).invoke(
        f"Answer based on:\n{docs}\n\nQuestion: {query}"
    ).content
Glean = enterprise search index. Deep Agent = reasoning engine on top.

Summary

OpenCode → Deep Agent: same LLM gateway credentials, new orchestration layer — persistent state, visible planning, VFS, specialised models, HITL, platform-deployable.
VFS is the key enabler: files never enter the context window — only paths do. Agents stay sharp across arbitrarily long tasks.
Skills bus: one make_llm(model) call in config.py controls every skill. Swap models or gateway without touching agent logic.
FastAPI → Aegra migration path: start raw, grow into the Platform API. Same agent code. SDK compatibility + LangFlow + Foundry unlocks.
No vendor lock-in: LangGraph (MIT) · Aegra (Apache 2.0) · FastAPI (MIT) · K8s-native · your LLM gateway, your models.
